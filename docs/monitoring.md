# Яндекс Monitoring в этом проекте — пошаговая инструкция

## Сначала важно

- **Я не могу зайти в твой Yandex Cloud, Railway или Vercel.** У меня нет твоих паролей и прав. Настройка = ты вставляешь данные в панели, я подсказываю по коду и по логам.
- **Что мне не присылай в чат:** приватный ключ целиком, `MONITORING_INGEST_SECRET` целиком, пароли. Это утечка. Можно: **первые 6–8 символов** ID, **текст ошибки** из `curl` или Railway, **скрин** (с замазанными секретами).
- **Что пришли, если хочешь, чтобы я «допинал» настройку по фактам:**  
  1) URL бэкенда на Railway (`https://…up.railway.app`)  
  2) Результат **self-test** (весь JSON ответа)  
  3) Одна строка из логов Railway: поиск по словам **`auth-api.boot`** или **`yc.monitoring`**  
  Без ключей и без PEM.

---

## Шаг 1. Консоль Yandex Cloud (один раз)

Сделай по порядку:

1. Зайди в [консоль Yandex Cloud](https://console.cloud.yandex.ru/).
2. Вверху выбери **каталог** (folder), где лежит проект. Запомни его — нам нужен **ID каталога**.  
   Где посмотреть ID: каталог → **Обзор** / настройки, поле вроде `b1g...` — это и есть **ID каталога**.
3. Создай **сервисный аккаунт** (или возьми существующий):  
   **IAM** → **Сервисные аккаунты** → создать → имя любое.
4. На этот аккаунт выдай роль на **этот каталог**: **`monitoring.writer`** (или шире — `editor`, если не найдёшь точную).
5. Зайди в **этот сервисный аккаунт** → вкладка **Ключи доступа** → **Создать новый ключ** → тип **Статический ключ доступа**.  
   Сохрани выданное:
   - **ID ключа** (Access Key ID)
   - **Секрет** — это и есть **приватный ключ в формате PEM** (блок с `BEGIN PRIVATE KEY` … `END PRIVATE KEY`).
6. Скопируй **ID сервисного аккаунта** (отдельное поле, похоже на `aje...`).

Итого у тебя на руках четыре строки:

| Что это | Куда потом в проект |
|--------|----------------------|
| ID каталога | `YC_FOLDER_ID` |
| ID сервисного аккаунта | `YC_SERVICE_ACCOUNT_ID` |
| ID статического ключа | `YC_ACCESS_KEY_ID` |
| PEM целиком | `YC_PRIVATE_KEY` или `YC_PRIVATE_KEY_BASE64` |

## Шаг 2. Railway — куда вставить переменные

Открой свой сервис на Railway (тот, где запускается **`node server/auth-api.mjs`**).

Добавь переменные окружения:

| Переменная | Значение |
|------------|----------|
| `YC_FOLDER_ID` | ID каталога из шага 1 |
| `YC_SERVICE_ACCOUNT_ID` | ID сервисного аккаунта |
| `YC_ACCESS_KEY_ID` | ID статического ключа |
| `YC_PRIVATE_KEY` | Весь PEM **одной строкой**: между строками PEM вставь символы **`\n`** (не настоящие переносы, а буквально слэш и n). Или см. ниже про Base64. |
| `YC_DEPLOY_ENV` | Например `production` |

Если Railway «ломает» ключ:

- Положи ключ в **`YC_PRIVATE_KEY_BASE64`**: закодируй весь файл PEM в **Base64 одной строкой** (без переносов). В коде это уже поддержано.

Затем нажми **Redeploy** приложения.

## Шаг 3. Секрет для проверки (обязательно для теста)

Чтобы никто чужой не дергал тестовый URL:

1. Придумай любую длинную случайную строку (например 32 символа).
2. В Railway добавь **`MONITORING_INGEST_SECRET`** = эта строка.

Тот же секрет понадобится только в заголовке запроса в шаге 4.

## Шаг 4. Проверка

### Почему было `{"message":"Не найдено","code":"not_found"}`

Так отвечает приложение, если запрос пришёл **не на тот адрес или не тем методом**. Чаще всего:

- Открыли в браузере **только домен** или **не тот путь** — нужен полный путь **`/api/monitoring/self-test`**.
- Раньше в коде был **только POST**, а браузер шлёт **GET** → получался 404 «Не найдено». Сейчас поддержаны **и GET, и POST**.

### Вариант А — браузер (быстро)

Подставь свой секрет из Railway (`MONITORING_INGEST_SECRET`) и открой одну строку в адресной строке:

```text
https://ТВОЙ-ДОМЕН.up.railway.app/api/monitoring/self-test?secret=ТВОЙ_SECRET
```

Секрет в URL виден в истории браузера — только для проверки, потом используй curl.

### Вариант Б — терминал (правильнее)

```bash
curl -sS -X POST "https://ТВОЙ-ДОМЕН.up.railway.app/api/monitoring/self-test" -H "x-monitoring-secret: ТВОЙ_SECRET_ИЗ_RAILWAY" -H "Content-Type: application/json"
```

**Если всё ок:** в ответе будет `"ok": true`, шаг `"write"`.

**Если не ок:** скопируй текст ответа целиком — по нему видно, ключ не тот (шаг `iam`) или доступ в Monitoring (шаг `write`).

### Где искать строку про Monitoring в Railway

После каждого деплоя в логах есть строка **`auth-api.boot`** с полем **`yandexMonitoringEnvOk`** (`true` / `false`).  
Дополнительно может быть **`yc.monitoring.ready`** или **`yc.monitoring.disabled`**.  

В панели Railway: сервис → **Deployments** → последний деплой → **View logs** → поле поиска: **`boot`** или **`monitoring`**.

## Шаг 5. Где в интерфейсе Яндекса увидеть метрику

1. [Консоль → Monitoring](https://console.cloud.yandex.ru/folders/) → выбери **тот же каталог**, чей ID в `YC_FOLDER_ID`.
2. Раздел метрик / обзор — выбери сервис **`custom`** (пользовательские метрики).
3. Ищи метрику **`saas.self_test.ping`** после успешного self-test.  
   Появление может занять **1–3 минуты**.

Дальше при работе сайта появятся свои имена: ошибки сервера — `saas.auth_api.error`, события — `saas.business.*` и т.д.

## Опционально: ошибки с сайта (Vercel → Railway)

Чтобы фронт слал ошибки на тот же бэкенд:

- В Vercel: `VITE_YC_MONITORING_INGEST=true`
- Тот же **`MONITORING_INGEST_SECRET`** на Railway и при желании **`VITE_MONITORING_INGEST_SECRET`** на Vercel (одинаковые значения).

---

## Отладка по логам

В Railway включи **`YC_MONITORING_DEBUG=1`** — в лог попадёт ответ API при каждой записи метрики. Только не оставляй это навсегда в проде без нужды.

---

## Частые ошибки (коротко)

| Что видишь | Что сделать |
|------------|-------------|
| В логах `yc.monitoring.disabled` | Проверь, что все четыре `YC_*` заданы и PEM начинается с `-----BEGIN` |
| Ответ self-test, шаг `iam` | Неверная тройка: аккаунт / ID ключа / PEM не от этого ключа |
| Ответ self-test, шаг `write`, 403 | На сервисный аккаунт не выдана роль **`monitoring.writer`** на каталог |
| Self-test ок, в графике пусто | Другой каталог в консоли; подожди 2–3 минуты; фильтр **custom** |
