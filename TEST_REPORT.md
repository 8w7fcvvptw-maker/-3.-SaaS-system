## 1. Безопасность и архитектура данных

**Аутентификация:** используется только **Supabase Auth** (сессия и JWT выдаёт Supabase). Отдельный самописный JWT, серверный middleware под выдачу токенов и ручное хеширование паролей в проекте **не реализуются** — этим занимается Supabase.

**Основная изоляция данных:** **RLS (Row Level Security)** в PostgreSQL: PostgREST отсекает строки по политикам. В JS повторяются проверки (например, `business_id`, валидация полей), чтобы возвращать **понятные сообщения** (400/403) до или вместе с ответом API.

**Клиент:** в формах есть **валидация на стороне браузера**; секреты в репозиторий не попадают — в `frontend/.env.example` только плейсхолдеры и комментарии, без реальных ключей и паролей.

### Цепочка запроса (кратко)

1. Браузер держит сессию Supabase; **Supabase JS SDK** подставляет **access token** в заголовки к PostgREST.  
2. **PostgREST** выполняет запросы к таблицам от имени роли `authenticated` / `anon`.  
3. **RLS** на строках решает, что пользователь может читать/писать.

**Гостевая онлайн-запись при строгих политиках RLS:** анонимный клиент не «видит» чужие данные, но вставка записи от гостя может потребовать отдельных политик или **Edge Function** с **service role** (вне браузера), иначе гость не сможет создать запись без нарушения модели доступа. Это ограничение модели, а не баг UI.

## 4. Ручная проверка для сдачи (компактный чеклист)

| # | Проверка | Ожидание |
|---|----------|----------|
| 1 | Отправить форму с **пустыми обязательными полями** | **400** / сообщение валидации (клиент + сервер при необходимости) |
| 2 | Зайти в кабинет **без входа** (или после выхода) | Нет данных дашборда или **редирект** на логин |
| 3 | Запрос с **чужим `business_id`** (подмена в API/консоли) | **403** / пустой результат / отказ RLS |

### Таблица «действие → ожидание → факт»

Заполните колонку «Факт» при демонстрации (достаточно ✓ / краткая пометка).

| № | Действие | Ожидание | Факт |
|---|----------|----------|------|
| 1 | Пустые обязательные поля в форме (клиент, услуга и т.д.) | Ошибка до запроса или ответ **400** с понятным текстом | |
| 2 | **Выход** из аккаунта / очистка сессии, обновление страницы кабинета | Нет доступа к данным бизнеса или редирект на вход | |
| 3 | Попытка доступа к данным с **не своим** `business_id` | **403**, пустой список или отказ на уровне БД (RLS) | |
| 4 | Успешный вход под владельцем бизнеса | Списки клиентов/услуг/записей соответствуют своему `business_id` | |
| 5 | Создание записи в сценарии бронирования (если есть гость) | Поведение согласовано с политиками RLS (см. раздел 1) | |

При сдаче можно приложить **1–2 скриншота**: например, форма с ошибкой валидации и экран после выхода / блокировки доступа.

## 5. Отладка окружения (одна фраза)

Если что-то «не цепляется» к Supabase: проверьте **`.env`**, **перезапуск dev-сервера** после смены переменных; иногда мешают **антивирус, корпоративный прокси или VPN**, режущие запросы к облаку.

---

## 6. Замечания по схеме Supabase

- Таблицы `clients`, `staff`, `services` часто требуют `business_id` (NOT NULL). В формах передаётся `business_id` из `getBusiness()`.
- Если в `businesses` нет строк, часть тестов создания пропускается — добавьте seed (см. `supabase/seed.sql` или вставку вручную).
- Имя колонки у сотрудников: в схеме может быть `service_ids` вместо `services` — при расхождении возможна ошибка при сохранении.

## 9. Рекомендации

1. Держать **seed-данные** в Supabase для стабильных интеграционных тестов.  
2. Политики **RLS** держать в актуальном состоянии вместе с миграциями (см. `supabase-rls-auth-migration.sql` при необходимости).  
3. При **UNIQUE** (например, телефон) — обрабатывать конфликт в UI и в ответах API.

## 10. Про интерфейс кабинета

Страницы **«Уведомления»** и блок **«Популярные услуги»** на **«Аналитике»** помечены в UI как **заглушка / демо** (нет реальной отправки SMS/Email; полосы популярности не считаются из агрегатов БД). Остальные разделы опираются на API и RLS.
